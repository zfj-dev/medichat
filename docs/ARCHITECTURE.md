# MediChat 架构设计

## 系统概览

MediChat 是一个基于 RAG 的中文医疗科普问答系统。以默沙东诊疗手册为知识库，结合混合检索、三层安全防线和 QLoRA 微调的 Qwen2-7B 模型，提供安全、可溯源的医疗知识回答。

## 核心设计决策

### 1. 为什么用混合检索而非纯向量检索？

**问题：** 医学术语高度精确（如"二尖瓣脱垂"），纯语义向量可能把"二尖瓣关闭不全"排在更前面，因为语义相近但并非同一概念。

**方案：** BM25 稀疏检索补充关键词级精确匹配，BGE 稠密向量处理语义泛化查询。两者加权融合（默认 0.7:0.3）后经 BGE-reranker 精排，兼顾精确和泛化。

**面试话术：** "我理解向量检索在专业领域的局限性——语义相似不等于内容相关。所以用混合检索补偿，BM25 做精确的词项匹配，稠密向量做语义扩展，reranker 做最后的精排。"

### 2. 为什么用三层安全防线而非单层 LLM？

**问题：** 单靠 LLM 做安全判断有三个缺陷：延迟高（每次多一次推理调用）、可控性差（prompt 可被 jailbreak）、成本高。

**方案：**
- 第一层（关键词规则）：最快，0 延迟，拦截直接的危险请求
- 第二层（BERT 分类器）：语义级判断，识别变体表述，argmax 直接判断不用阈值
- 第三层（输出后过滤）：最后防线，追加免责声明，过滤危险输出

**面试话术：** "分层架构的核心思想是每层处理自己能高效处理的复杂度。关键词做快筛，BERT 做语义理解，后过滤做兜底。这比用一个 LLM 做所有判断更可靠、更快。"

### 3. 为什么用 QLoRA 而非全量微调？

单卡 3080 20GB 可以跑 7B 模型的 QLoRA 微调，全量微调需要至少 4 张 A100。对于让模型学会拒绝诊断、主动加免责声明这类行为对齐任务，LoRA 的低秩适配完全足够。

## 数据流

1. 用户输入 → FastAPI `/chat`
2. 安全检测：关键词 → BERT → 判断 normal/emergency
3. 正常查询 → 混合检索（BM25 + BGE → 融合 → Reranker）
4. Top-K 文档 + 系统提示 → Qwen2-7B (LoRA) 流式生成
5. 输出后过滤 → 添加免责声明 → 返回用户

## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 大模型 | Qwen2-7B-Instruct | FP16, LoRA 微调 |
| 向量模型 | BGE-large-zh-v1.5 | 中文向量化 |
| 重排序 | BGE-reranker-v2-m3 | 精排 |
| 向量数据库 | Milvus Lite | 嵌入式向量存储 |
| 后端框架 | FastAPI | SSE 流式输出 |
| 前端 | Streamlit | 聊天界面 |
| 安全分类 | BERT (transformers) | 文本分类 |
| 微调 | PEFT (LoRA) | QLoRA 高效微调 |
| 部署 | Docker Compose | 一键部署 |
